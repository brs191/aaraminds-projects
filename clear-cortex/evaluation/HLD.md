## 2 · Purpose & context

The Credit Routing Service ("CLEAR" — Credit Logging and Evaluation Assistance Repository) is AT&T's **decision-and-routing service for business credit checks**. It accepts credit-check requests, applies finance-owned configurable rules to decide which downstream credit engine and policy apply, dispatches to that backend, persists the decision, and publishes it as an event for the order lifecycle. Primary callers are a CLEAR web front-end (v2) plus legacy v1 front-ends, internal services, and credit-finance admins; the routing *targets* are Equifax (via the CSI SOAP ESB and direct UBCT/ICAAM) and the CAS REST gateway (UCCS + CSRM). *Evidence: `Code_Briefing.md` §1; `Credit.yaml` title; `README.md` L5-7.*

## 3 · Scope

Whole single-module Spring Boot service, at component altitude. **Covered:** the routing/decision core, the full REST surface (v1 + v2 + admin + internal), the MongoDB data model, the five external integrations, and the cross-cutting mechanisms (§§4–10) — all six high-value subsystems were deep-read to file:line in P2. **Not visible in scoped code:** the metrics/log collection backend (OTel exporter / Grafana / Prometheus / Sentry are not wired in code — assume agent/infra-side); the internals of the external credit engines (CSI / CAS / Equifax); and the **out-of-band audit writer** (the CDC/change-stream/Atlas trigger that populates the `*_audit` collections, which lives outside this repo). **Out of scope by method:** the runtime/production-data layer.

## 4 · Architecture overview

The service is a single Spring Boot 3.3.9 / Java 17 application, layered **controller → processor → routing → strategy → backend client**, over MongoDB (persistence), Kafka (eventing), and SOAP/REST egress to credit bureaus. Its defining shape is a **rules-driven router**: an inbound request is reduced to a parameter map, a DSL rule set stored in MongoDB (read fresh per request — the routing lookup is not cached) is evaluated to a `CCTarget{system, api}`, and a strategy + factory pair dispatches to the named backend (CSI / CAS / Equifax, or an internal auto-approve / pre-approval / govt-exempt path). A request-scoped object, `CreditCheckRequestScope`, threads state through the call as a shared blackboard, and a thick layer of **AOP aspects** owns persistence, audit, and status transitions *around* the service methods rather than inside them. v2 extends v1 with multi-product evaluation (mobility + wireline in parallel) and identity verification. See `../design/architecture-component-view.svg` for the component view and `../design/credit-check-v2-runtime-flow.svg` for the runtime flow. *Evidence: `Code_Briefing.md` §2, §6, §8.*

## 6 · Domain & data model

The model is **29 MongoDB collections** (from 30 `@Document` classes, one of which — `AuditableEntity` — is a mapped superclass with no collection of its own), with **no relational store and no JPA associations**. The spine is the **credit-check result**: `creditCheckResult` (`@Id creditCheckProcessId`) is 1:1 with `creditCheckDetails` (the same PK, set in `CreditCheckProcessorAspect`) and rolls up N:1 into `multiProductCreditCheckResult` (`@Id creditProcessNumber`); `creditCheckParameter` and `ubctTransactions` carry the same keys for monitoring and bureau-transaction tracking. The **routing rule** (`cCRule`, `@Indexed creditCheckType`) embeds the `Rule<CCTarget>` `when/then` definition and has an `ACTIVE/INACTIVE/DELETED` lifecycle (soft delete). Around these sit reference-data collections (config, product/policy/account-type/EIP/pre-approval/BCES/SAART mappings), RBAC (`roles`, `userroleassignments` — referenced by role *name*), one analytics snapshot collection, and ten generic `*_audit` collections that embed full document snapshots.

Relationships use exactly three mechanisms — a **shared-PK** spine (`creditCheckResult`/`creditCheckDetails`), **denormalized FK-by-scalar** children (via a copied `creditProcessNumber`), and embedded value-objects, plus **code-field reference lookups** into the mapping collections — with **no DBRefs / object references** anywhere. The operational risk is indexing: only **3 declared indexes** exist service-wide *and* `spring.data.mongodb.auto-index-creation` is **off**, so the hot collections are effectively `_id`-only — every admin search/export (an unanchored `$regex` on a caller-chosen field), the `/trace` reconciliation, and the nightly stats schedulers full-scan unbounded collections that have **no TTL/purge**, risking RU-throttling that bleeds into the live write path on a throughput-priced datastore (see §11(5)). *Evidence: `Code_Briefing.md` §3, §12.*

## 7 · Key runtime flows

**v2 single-product credit check.** The controller seeds `CreditCheckRequestScope`; a processor aspect hydrates and persists a draft `CreditCheckResult` and buffers the payload; `CreditCheckProcessor` runs the enricher chain, then `RoutingService.determineRoutingBackendSystem` projects the request into a `CCRequestRule`, flattens it to a parameter `Map`, loads the **ACTIVE** rules for the `creditCheckType` from Mongo (`getCCRuleByCCType` — **read fresh per request, not cached**), and evaluates them first-match-wins via `CCRuleExecutionService.executeRule` → a `CCTarget{system, api}`. A route-specific `CustomValidator` runs, then `CreditCheckServiceFactory` resolves the concrete `CreditCheckService`, which calls the chosen backend (CSI / CAS / Equifax) and maps the result back onto the scope; a closing aspect persists the result + details + the monitoring trail and stamps the `creditProcessNumber` on the response. `PreApproval` short-circuits to a local decision before the rule engine.

**Dispatch.** The resolved `CCTarget` drives `CreditCheckServiceFactory` to one of **7 routable strategies** — 4 CSI (ECC / AddAccount / ESOCC / EUCC) and 3 Local (GovtExempt / AutoApprove / PreApproval); only 7 of `CCTargetAPI`'s 13 values are routes (the rest are refresh-path/debt-sub-call tags, and `EIP` is a dead enum). Three other sites reuse the rule engine: v1 (`CCRoutingService`, inline `switch` dispatch); a diagnostic `/v2/internal/.../routing-rule` (`getMatchedRule`, returns *all* matching rules, never throws); and **UBCT** (`deriveRequestType`, which uses the *second* evaluator `admin/rules/RuleExecutionService` and **swallows** a no-match to `null`). Error asymmetry: no ACTIVE rule for a type → **404**; rules exist but none match → **`RuleNotMatchedException` → HTTP 500** (no handler); UBCT → `null`/`REQUEST_TYPE_ERROR`.

**Multi-product.** The request splits into Wireless + Wireline children (shared CPN) run in parallel on a `CompletableFuture` pool joined **fail-fast** (`allOfFailFast`): a single failing leg aborts the whole transaction → status `FAILED`, so there is **no partial-approved aggregate**; when both succeed the overall status is a rank-reduce (most-restrictive leg wins). The aggregate status is a **free-form String** managed entirely by a 796-line `MultiProductCreditCheckAspect`; only the POST path forces `FAILED` on error (refresh/patch/cancel leave it unchanged).

**Eventing.** Both create happy paths (single- and multi-product POST) are **publish-silent**; a credit decision reaches Kafka only on specific status-change paths — ESOCC refresh, UBCT refresh, and the scheduled BCES retry. On multi-product refresh the publish happens in the processor method body *before* the closing aspect's status save (an emit-before-persist gap — see §11(1)). *Evidence: `Code_Briefing.md` §10 (DSL), §11 (runtime), §6, §8.*

## 8 · External interfaces & integrations

**Inbound: 89 routable REST endpoints** under `/CreditCheck` — v1 (3), v2 single-product (4), v2 multi-product (8), v2 policy (3), v2 internal (2), admin (62) + ubct config (3), internal/ops ping + cache (4). (Counting convention: one canonical path per method plus the 3 ActuatorController ping paths, excluding the v2 dual-mount aliases — which bring it to 104, live on 3 controllers.) The published `Credit.yaml` contract covers only **6 of 89** (the v2 single-product happy path + cancel + ping) and **omits the entire multi-product + policy surface** its own SpringDoc groups expose, and lacks a server/base-path — a documentation-drift finding; treat the SpringDoc runtime groups as authoritative.

**Outbound: three backends, three transports.** **CSI SOAP → Equifax** via the ESB — low-level SAAJ (`SoapCallService` builds a raw `SOAPMessage`, not a JAX-WS port), **9 operations** of which 4 are primary credit-check routes (ECC / AddAccount / ESOCC / EUCC, matching the 4 CSI strategies in §7) and 5 are debt/refresh/cancel sub-calls. **CAS REST → UCCS** (identity/credit: SUCA / IUCVQ / VUCVA; IUCAD-on-UCCS is unused) **+ CSRM** (policy: EUCPC / IUCP / UUCP) over generated OpenAPI clients consumed **models-and-Basic-auth-only**. **Equifax-direct → UBCT** (legacy `/attgbs/`, Customer/Security headers) **+ ICAAM** ("UBCT2.0", canary-gated by a `creditCheckId` regex, bearer) — the UBCT submit is `@Async` and poll-decoupled. **Kafka** publish → `…creditcheckupdates`, consume → SAART segments.

**Auth dependencies.** Microsoft Entra ID underpins the inbound resource-server JWT and the Kafka OAUTHBEARER token; the **ICAAM downstream token is issued by Equifax's own OAuth2** (`api.uat.equifax.com/v2/oauth/token`, client-credentials — *not* Entra); opaque-token introspection uses the AT&T eLogin IdP (`oidc.stage.elogin.att.com`). The policy/order-update endpoints are served by a **hand-written** `CreditPolicyController` over the generated `creditPolicy` **models** — the generated `CreditApi` server interface is **unused** (0 source refs).

**Two integration-depth findings.** CSI message "security" is a **plaintext username/password in the SOAP header with no XML signing** — message-level confidentiality rests on TLS alone and there is no message integrity (extends §11(6) to the wire). And the synchronous REST downstreams (CAS + the in-request CSI legs) have **no circuit breaker, bulkhead, or connection-pool ceiling** and a 60 s read timeout — a cascading-failure exposure (§10, §11(9)). *Evidence: `Code_Briefing.md` §13, §4, §5.*

## 10 · Key design decisions & patterns

- **MongoDB document store, no transactions.** *Observed:* `@Document` model, zero `@Transactional`. *Evidence:* `Code_Briefing.md` §3, §6. *Likely rationale (`inferred`):* schema flexibility + horizontal scale for a high-volume decision/audit store. *Trade-off:* no atomicity across the multi-write result + details + monitoring sequences.
- **DSL rule engine, externalized — but duplicated and divergent.** *Observed:* a `when/then` `Rule<CCTarget>` model in `cCRule`, evaluated by **two forked engines** — `CCRuleExecutionService` (routing, type-agnostic) and `RuleExecutionService` (UBCT, type-strict, +4 operators); the routing lookup reads Mongo **per request** (the `@Cacheable` is on methods routing never calls). *Evidence:* `Code_Briefing.md` §10 — `gt` = `>=` at `CCRuleExecutionService › L132-134` vs strict `>` at `RuleExecutionService › L183`. *Likely rationale (`inferred: high`):* change routing logic without redeploying; the second engine grew independently for UBCT. *Trade-off:* the two are **not interchangeable** (same operator strings, different results) — consolidation is a breaking migration, and the divergence is **unguarded by tests**.
- **AOP-as-persistence.** *Observed:* aspects own result persistence, status transitions, and the monitoring trail. *Evidence:* `Code_Briefing.md` §6, §11. *Likely rationale (`inferred`):* keep processors decision-focused, centralize audit. *Trade-off:* heavy coupling to `CreditCheckRequestScope` + non-obvious control flow (the multi-product state machine is a single 796-line aspect).
- **Strategy + factory routing dispatch.** *Observed:* `CCTarget{system,api}` drives factories → concrete backend strategy. *Evidence:* `Code_Briefing.md` §2, §8, §11. *Rationale (`inferred`):* open/closed addition of new credit backends. *Trade-off:* indirection between request and the backend actually called.
- **Kafka eventing via a hand-rolled wrapper.** *Observed:* `MessageBrokerClient` wraps a raw `KafkaProducer` (not `KafkaTemplate`); publish only on select status-change paths. *Evidence:* `Code_Briefing.md` §5, §8, §11. *Rationale (`inferred`):* fine control over the IEBus event contract. *Trade-off:* bypasses Spring Kafka conveniences; publish coverage is uneven.
- **Fail-fast parallel multi-product, no transactional outbox.** *Observed:* Wireless + Wireline legs run on `CompletableFuture`/`allOfFailFast`; the aggregate status is a rank-reduce over the legs; the refresh event is emitted *before* the status save. *Evidence:* `Code_Briefing.md` §11. *Rationale (`inferred: high`):* leg-parallel latency + a simple single-topic event contract. *Trade-off:* cannot represent partial approval (one failing leg fails all); and emit-before-persist with no outbox means a published event can announce a status the DB never recorded.
- **Synchronous downstream calls with no fault-tolerance layer.** *Observed:* every credit-bureau/gateway call (CSI SOAP, CAS UCCS/CSRM, Equifax UBCT/ICAAM) is a blocking REST/SOAP call over `SimpleClientHttpRequestFactory` (no connection pool), with **no circuit breaker, bulkhead, or rate limiter** and retry only on the CSI leg; downstream errors are caught and mapped to a uniform `ErrorResponse`. *Evidence:* `Code_Briefing.md` §13. *Likely rationale (`inferred: high`):* simplest possible client wiring; the `@Async`/poll decoupling of UBCT was judged sufficient and bureau calls assumed reliable. *Trade-off:* a **slow-but-up** downstream (the classic bureau brownout) holds threads the full 60 s with no pool ceiling to fail fast against and no breaker to trip — `CallerRunsPolicy` then spreads the stall to caller threads ⇒ thread/connection exhaustion. Stack-consistent remedy: Resilience4j (`CircuitBreaker`+`Bulkhead`+`TimeLimiter`) + a pooled HttpClient 5 factory.
- **Authentication at the edge; authorization delegated to the client.** *Observed:* a multi-issuer OAuth2 resource server verifies tokens, but every endpoint resolves to `.authenticated()` and the RBAC model is computed only at login and returned to the SPA. *Evidence:* `Code_Briefing.md` §14 — 2 `.authorizeHttpRequests`, both `.anyRequest().authenticated()`; `getAllPermissions` flows into a `UserInfo` DTO, never a server gate. *Likely rationale (`inferred: medium`):* the SPA renders role-appropriate UI and was trusted to self-restrict; back-end authz was deferred. *Trade-off:* the server cannot enforce least privilege — any authenticated principal can call any admin/rules/role-grant endpoint directly; the RBAC store is world-writable to any valid token. Remedy: a back-end authorization layer (method security or a gateway PDP keyed on the existing `roles`/`permissions` data).

## 11 · Observations & remediation

Risks and debt, in rough severity order **(1) → (2) → (9) → (5) → (4) → (6) → (3) → (7) → (8)**. Each is concrete and code-anchored in `Code_Briefing.md`; the numbering is stable (referenced across the artifacts).

**(1) No transaction management — highest severity.** Zero `@Transactional` / `MongoTransactionManager` anywhere, so one request does 3–10+ unguarded Mongo writes (result → details → monitoring, plus async `ubctTransactions`, plus 10+ across 5 collections on multi-product) and a non-transactional Kafka emit. A mid-sequence failure leaves **orphaned results** (a decision row with no details/monitoring, shown as complete), **watermelon multi-product** aggregates (a parent set `FAILED` over an already-`APPROVED` child), or a **published event for a status the system of record never persisted** (the multi-product refresh emits before its final save, with no retraction event). `@AfterThrowing` is forward-repair, never rollback, and does not fire on `Error`/pod-kill. *Concrete in `Code_Briefing.md` §11.3.* **Remedy:** a transaction boundary (Mongo multi-document transactions or a saga) around the write sequence, and a **transactional outbox** so events never precede the persist they announce.

**(2) Authorization is authentication-only.** A whole-tree grep finds **zero** role/scope/method-security gates; both `.authorizeHttpRequests` branches terminate at `.anyRequest().authenticated()`. So any valid token of the right issuer reaches **every** protected endpoint — all rules CRUD and the **role-grant endpoints themselves** (the privilege store is world-writable to any authenticated caller). The rich RBAC model (44-permission enum, `roles` + `userroleassignments`, full CRUD) is computed once at login into a `UserInfo` DTO and **never read again** server-side — client-side authorization theater. *§14.3.* **Remedy:** enable method security or a gateway PDP keyed on the existing `roles`/`permissions` data.

**(9) No fault-tolerance layer on the synchronous downstreams.** No circuit breaker / bulkhead / rate limiter; an unpooled `SimpleClientHttpRequestFactory` with a 60 s read timeout; retry only on the CSI leg. A slow-but-up CAS or Equifax endpoint holds request threads the full 60 s with no pool ceiling to fail fast against, and `CallerRunsPolicy` spreads the stall to caller threads ⇒ thread/connection exhaustion on the in-request CAS + CSI legs (the UBCT submit is insulated by `@Async`/poll). *§13.5.* **Remedy:** Resilience4j (`CircuitBreaker` + `Bulkhead` + `TimeLimiter`) around the CAS + Equifax clients, and a pooled Apache HttpClient 5 factory with a bounded connection manager.

**(5) Effectively index-less hot collections.** Only **3 declared indexes** service-wide **and** `spring.data.mongodb.auto-index-creation` is off → the hot collections are `_id`-only. Admin search/export (an unanchored `$regex` on a caller-chosen field), the `/trace` reconciliation, and the **daily recompute scheduler** (which filters on an unindexed `updatedDate` while sorting on `createdDate`, then fans out 5× `computeAndSave` per back-dated record) all full-scan unbounded, never-purged collections — an availability risk on a throughput-priced (Cosmos-for-Mongo) datastore. *§12.2–§12.3, §15.2.* **Remedy:** declare the missing compound indexes **and** enable `auto-index-creation` (or ship an index migration); add TTL/retention to the result/monitoring/audit collections.

**(4) The two DSL evaluators are semantically divergent.** Not just duplicated — `gt` means `>=` in the routing evaluator but strict `>` in the admin/UBCT one, and 4 admin-only operators throw in routing. A rule authored and validated in admin can silently mis-route or fail in production, and the divergence is **unguarded by tests** (the one evaluator spec uses a single fixture whose every `then` is the same target). *§10.1.* **Remedy:** consolidate to one evaluator behind a shared contract with a differential test suite; treat the merge as a breaking migration of existing rules.

**(6) Secrets in clear, on the wire and at rest.** **6 fully-plaintext credential values** are committed as `application.yml` defaults (Mongo Atlas pw, CAS `Clear@AzureProd2025_2026`, Equifax ICAAM key + secret, CSI pw, Entra-Kafka `clientSecret`) — proven sensitive because the **test** yml masks or localizes the same fields. On the wire, CSI authenticates with a **plaintext username/password in the SOAP header with no XML signing**, and `TokenProvider` logs the raw ICAAM bearer token (`// NOSONAR`). *§13.2, §14.6.* **Remedy:** move all secrets to **Azure Key Vault via managed identity**; remove the token log; rely on TLS + (if CSI requires it) WS-Security signing.

**(3) Dead/buggy security components.** The opaque-token Caffeine cache is unwired → eLogin introspection runs **per request, uncached**; `CustomAuthenticationEntryPoint` is unwired (and weaker than the Spring default — delete, don't wire); `isJwtShaped` **ignores the token** and routes by the `AUTH_PROVIDER` env var, so one instance can't serve mixed JWT+opaque traffic; inbound JWT **`aud` is not validated**; and the Halo `/public/oidc` login branch `JWT.decode`s without verifying the signature. `/v2/internal/**`, `/public/oidc/**`, `/sync/saart-segments/**` are unauthenticated. *§14.2, §14.4, §14.5.* **Remedy:** wire the introspection cache; add an `aud` validator; verify the Halo token signature; delete the dead entry point.

**(7) `Credit.yaml` is a stale 6-of-89 subset** missing the multi-product + policy surface its own SpringDoc groups expose. *§4.* **Remedy:** regenerate the published contract from the runtime groups, or treat SpringDoc as the source of truth.

**(8) `CacheService.evictCache` uses a literal (non-SpEL) cache name** — the manual evict likely targets the wrong cache. *§11(8) origin in breadth.* **Remedy:** fix the cache-name expression (and note that the routing hot-path is uncached anyway, so this cache governs only the by-id admin GET).

**Modernization.** The repo's `.github/appmod/appcat` folder is present but **empty** — no Microsoft AppCAT scan output is committed — so there are no tool-generated modernization findings to fold in here. The priorities above are derived from this comprehension. Against the canonical AaraMinds/AT&T stack, the highest-leverage moves are: transactions + outbox (1), back-end authorization (2), Resilience4j + pooled HTTP factory (9), an index migration + retention (5), DSL-evaluator consolidation (4), and Key Vault for secrets (6).
