## 8 · Candidate design decisions (HLD §10 seed)

1. **MongoDB document store, no transactions** — `@Document` model, zero `@Transactional`/`MongoTransactionManager`. Trade-off: schema flexibility + scale vs. **no atomicity** across the multi-write result+details+monitoring sequences (which run inside AOP advice). `[deterministic on the absence; trade-off inferred]`
2. **Two duplicated rule evaluators — semantically divergent** — `routing/services/CCRuleExecutionService` (live, type-agnostic) vs `admin/rules/RuleExecutionService` (UBCT, type-strict, +4 operators). Same `Rule<T>` model; **`gt` means `>=` in routing but strict `>` in admin**, and admin-only operators *throw* in routing. `[deterministic]` — see §10 (P2 deep-read). Consolidation is a breaking migration, not a refactor.
3. **AOP-as-persistence** — aspects own result persistence + status transitions + the monitoring trail, keeping processors decision-focused. Deliberate, but heavy coupling + non-obvious control flow. `[deterministic]`
4. **Strategy + factory routing dispatch** — `CCTarget{system,api}` from the DSL engine drives `CreditCheckServiceFactory`/`CreditCheckRefreshFactory` → concrete CSI/CAS/local backend. The central routing contract. `[deterministic]`
5. **IEBus/Kafka eventing wrapper** — `MessageBrokerClient` wraps a raw `KafkaProducer` (not `KafkaTemplate`); publish only on specific refresh/UBCT paths, not the single-product happy path. `[deterministic]`
6. **v1/v2 split** — v1 self-contained inline dispatch (`CCRoutingService`); v2 factory-based, adds multi-product + identity. `[deterministic]`

## 10 · P2 deep-read — D1: DSL rules engine (2026-06-01, against `e17fe410`)

### 10.1 Two evaluators — duplicated AND semantically divergent `[deterministic]`
Both `routing/services/CCRuleExecutionService` (live routing) and `admin/rules/RuleExecutionService` (UBCT) evaluate the same `Rule<T>` JSON DSL (`when` map AND-combined, first-match-wins, `RuleNotMatchedException` on no match). Silent, no-error routing difference. **Highest-severity divergence.**

### 11.2 Multi-product state machine (the P1-deferred piece) `[deterministic]`
Status is a **free-form `String`** (`MultiProductCreditCheckResult.status L27`; the `MultiProductCCStatus` enum exists but is **unused**). States: `DRAFT` → `{APPROVED | FAILED | CANCELLED | …frontEndStatus}`. The machine lives entirely in **`MultiProductCreditCheckAspect`** (796 lines) — ~20 transition points across process/refresh/patch/cancel/IUCVQ/VUCVA. **Key asymmetry: only the POST-create path forces `FAILED` on exception (`L410`); refresh/patch/cancel/VUCVA failure advice leave the status UNCHANGED.** Cancel is guarded to `Approved`/`ACTIVE` only (`L471-481`); two `// TODO: revisit status` sit in the success advice.
**Parallel legs:** the request splits into Wireless + Wireline children (shared CPN), run in parallel via `CompletableFuture.supplyAsync` on the `@Primary threadPoolForAsyncMethod` pool, joined by **`FutureUtils.allOfFailFast` — fail-fast: the first leg to throw aborts the aggregate and the sibling's result is discarded.** ⇒ **there is no "one approved / one failed" persisted aggregate on POST; a single failing leg → the whole transaction is `FAILED`.** When both legs succeed, the overall status is a **rank-reduce — the most-restrictive leg wins** (`MultiProductMapper.mapToMultiProduct L117-181`; Approved=1 … FAILED=10). `productDetails` attach only on `Approved`. Partial-cancel: a still-`Approved` surviving leg keeps the aggregate `Approved` (`L810-847`).

### 11.3 §11(1) atomicity — made concrete `[deterministic]`
**No transaction boundary anywhere** (grep: 0 matches for `@Transactional` / `MongoTransactionManager` / `TransactionTemplate` / `ClientSession` / `withTransaction`). One single-product request = **3 sequential unguarded writes** (`creditCheckResult` → `creditCheckDetails` → `creditCheckParameter` monitoring, inside `CreditCheckProcessorAspect` advice) **+ 1..N concurrent `ubctTransactions` writes on an `@Async` thread**; one multi-product request = **10+ writes across 5 collections + a non-transactional Kafka emit**. The monitoring trail is buffered in the request-scoped bean and flushed once at end. Top failure modes (no rollback, no outbox, no saga):
- **Orphaned result** — W1 commits `creditCheckResult`, W2 (`creditCheckDetails`) throws → a decision row with no details and no monitoring trail; admin/export show it complete (S1; S4 on pod-kill, *no error logged*).
- **Watermelon multi-product** — a child fails after a sibling committed `APPROVED`; the parent is set `FAILED` (`L410`) over an `APPROVED` child; `productDetails` linkage lost (S6).
- **Event without persist** — on multi-product refresh the Kafka event is emitted in the advised method body — `MultiProductCreditCheckProcessorImpl#refreshMultiProductCreditCheck L545` — **before** the aspect's `@AfterReturning` final status save (`MultiProductCreditCheckAspect#afterRefreshMultiProductCreditCheck L373`); if the save then throws, downstream consumers act on a status the system of record never persisted — **no retraction event** (S7). (Note: this is a processor-method publish, not an `@Around`/`proceed()` — the multi-product aspect has no `proceed()`.) Highest impact — it escapes the service boundary.
- **UBCT decoupling** — the `@Async` `ubctTransactions` writes can orphan a `PENDING` row; the UBCT poller is the only compensation and heals only `ubctTransactions` status, not cross-collection consistency (S5).
`@AfterThrowing` is forward-repair (writes *more* rows), never rollback, and does not fire on `Error`/kill.

### 11.4 Eventing — precise `[deterministic]`
Single topic `com.att.clear.dev.creditcheckupdates` (`iebus.topic.interface`), record key always `"CreditCheckResponse"`, raw non-transactional `KafkaProducer.send` (`MessageBrokerClient.clientProducer L41-76`). Publishes occur on exactly: (a) **multi-product refresh** with an ESOCC `creditCheckInternalStatus` change (HTTP refresh or the UBCT poll); (b) **single-product UBCT-route refresh** with a status change; (c) the **scheduled BCES retry**. **Both happy paths (single- and multi-product create/POST) are publish-silent** — confirms + sharpens the P1 claim.

### 11.5 Blackboard + off-thread binding `[deterministic]`
`CreditCheckRequestScope` (`@RequestScope`, 12 fields) threads identity/route/result/monitoring/flags through the flow; controllers seed it, aspects mutate it. Background callers (scheduled retry, UBCT poll, the parallel legs) install a hand-rolled `CustomRequestScopeAttr` (HashMap-backed `RequestAttributes`) via `RequestContextHolder` + reset in `finally`. ⚠ **Gap:** the `@Async` `UBCTService.processUBCTRequest` and `CCResultMonitoringService` are NOT bound — they avoid resolving the scoped bean, but any transitive scope read on those threads would fail.

### 11.6 Test oracle `[deterministic]`
The flow/state-seeding aspects + the per-op happy/failure paths are reasonably covered (Spock + ITs). **Untested:** the multi-product **publish-decision true-branch** (ESOCC refresh status-change → `checkIfPublishRequired`/the `L545` publish has zero coverage); **parallel-leg partial-failure** (one leg commits while the sibling throws); the **off-thread scope binding under real concurrency** (the executor is mocked to run on the test thread); and the **no-transaction partial-write** behavior. `MessageBrokerClient.clientProducer` has no direct test.

### 11.7 D2 verdict
The runtime flow is a rules-driven, strategy-dispatched, aspect-persisted pipeline whose correctness rests on conventions the type system doesn't enforce: a free-form status String, fail-fast parallel legs that can't represent partial approval, persistence + state transitions scattered across a 796-line aspect with no transaction boundary, and a Kafka emit that can precede the persist it announces. The §11(1) risk is now concrete and the highest-severity item in the comprehension — and none of its load-bearing failure modes is guarded by a test.

### 13.5 ⚠ Resilience posture — synchronous + no circuit-breaking ⇒ cascading-failure exposure `[deterministic facts; inferred: high assessment]`
**There is no circuit breaker, bulkhead, or rate limiter anywhere** — grep across `src/main` + `pom.xml` for `Resilience4j | CircuitBreaker | Hystrix | Bulkhead | RateLimiter | Sentinel | failsafe` returns **zero** (the lone `failsafe` hit is the Maven *maven-failsafe-plugin*). The shared `RestTemplateConfig › L16-27` uses `SimpleClientHttpRequestFactory` — **JDK `HttpURLConnection`, no connection pool / no max-conn cap** — with read **60 s** / connect **1 s** (`yml › L293-295`). Three independent client beans (the primary `RestTemplate` → UBCT/ICAAM/TokenProvider; `ApiClientConfig` → CAS; `UnifiedPolicyConfig` → CSRM) all share those timeouts. `CustomResponseErrorHandler#hasError › L18-25` deliberately returns **false for 4xx/5xx** (don't-throw; `UBCTService` branches on status). **Retry exists in exactly one place** — the CSI `@Retryable` (§13.1, 3×/2 s, gated by `retryEnabled`); UBCT/ICAAM/TokenProvider/CAS get **no** in-call retry. The generated CAS/CSRM `ApiClient` has a retry loop but `maxAttemptsForRetry` **defaults to 1 and is never overridden** ⇒ effectively no retry. The only concurrency bounds are two `ThreadPoolTaskExecutor`s (`TaskExecutorConfig` — scheduler 3/5/5, async 10/20/50, both **`CallerRunsPolicy`**), which are not per-downstream bulkheads.

**Assessment `[inferred: high]`:** the error-*mapping* layer is unified and clean (`RemoteServiceException → 500` CSI · `CASResponseException → ex.httpStatus` CAS · `ApplicationException` ICAAM, all → one `ErrorResponse`), but the error-*tolerance* layer is essentially absent for the REST downstreams. The dangerous case is **slow-but-up**: a TCP-accepted-then-stalled Equifax/CAS endpoint holds the calling thread the full **60 s** with no max-conn cap to fail fast against and no breaker to trip — under load this is textbook thread/connection exhaustion, and `CallerRunsPolicy` then runs overflow on the **caller** thread, spreading the stall. **Hard-down is survivable** (connect fails at ~1 s; `ApiService` maps `UnknownHostException → 503`; UBCT just records `FAILED` and the cron re-submits). The genuinely in-request synchronous legs are **CSI/SOAP** and **CAS** (`fetchIUCVQ` → UCCS, no breaker); the **UBCT submit is insulated** (it's `@Async` + poll-decoupled). Stack-consistent fix: Resilience4j `CircuitBreaker`+`Bulkhead`+`TimeLimiter` around the CAS + Equifax clients, and swap `SimpleClientHttpRequestFactory` for a pooled Apache HttpClient 5 factory with a bounded connection manager.
