> ## ⚠️ CORRECTION (2026-06-17) — CLI and IDE are SEPARATE local sources
>
> The "Context" and "Decision §1" below assert that **CLI and IDE write to the same event schema /
> same stream** and therefore reject separate readers (Alternative 1). **That premise is empirically
> wrong** and is hereby corrected. It came from `IDE_USAGE_FINDINGS.md`, which was generated on a
> machine that has the Copilot **CLI** installed — so every session it saw was a CLI session. An
> engineer who uses **VS Code Copilot Chat only** (no CLI) shows **zero** data, because:
>
> | | Copilot CLI | VS Code Copilot Chat |
> |---|---|---|
> | Local store | `~/.copilot/session-state/<uuid>/events.jsonl` (+ `session-store.db`) | VS Code user data: `…/User/workspaceStorage/<ws>/chatSessions/`, legacy `…/GitHub.copilot-chat/transcripts/*.jsonl`, `…/globalStorage/emptyWindowChatSessions/` |
> | Producer | `copilot-agent` | VS Code Copilot Chat extension |
> | Cost unit | nanoAIU → credits (authoritative) | **token counts only** → credits *estimated* via price table |
> | Captured today | ✅ yes | ❌ **no** |
>
> **Corrected decisions:**
> 1. **Alternative 1 ("Separate CLI and IDE Readers") is now ACCEPTED, not rejected** — they are
>    genuinely different sources at different paths with different schemas. There is no shared
>    `events.jsonl` for IDE Chat.
> 2. The current `ideCollector` implementation is **wrong**: it re-reads `~/.copilot/session-state/`
>    and filters on an unverified `vscode.metadata.json` marker. It must instead read the VS Code
>    Chat paths above. Until rewritten, IDE usage is **not** captured.
> 3. **Cross-source dedup-by-ID is fine and mostly defensive** — CLI and IDE session IDs come from
>    different products and will not collide, so merging them does not erase either. (The earlier
>    same-directory design *did* erase the IDE source on ties; that goes away once the IDE reader
>    points at VS Code storage.)
> 4. **Open question:** whether VS Code Chat persists token data locally at all (vs. server-only).
>    Resolve by running `discover-ide-usage.sh`/`.ps1` (now probes the Chat paths) on an IDE-only
>    machine. If Chat data is server-only, IDE capture requires GitHub's billing/usage API — a
>    **network** source that would amend ADR-001.
>
> Everything below about **within-a-CLI-session** dedup (eventId, apiCallId, per-turn vs. shutdown
> reconciliation, prefer-shutdown precedence) remains valid for the CLI source and is unaffected.

## Context

### Current State
The Copilot Agent (CLI-based) emits telemetry to `~/.copilot/session-state/{sessionId}/events.jsonl` in JSONL format. Each session produces:
- Multiple `assistant.message` events (one per model invocation), each with granular token counts
- One `session.shutdown` event (at the end), with aggregated usage metrics per model

Session metadata resides in `~/.copilot/session-store.db` (SQLite), with tables `sessions` and `turns` providing context (cwd, repository, branch, timestamps).

### Unified Source Today
- Both CLI and IDE write to the same event schema (no `host_type` or environment marker in events)
- Future: IDE integration may add separate signals, but today they are structurally identical
- No existing separation logic required

### Double-Counting Risks

1. **Same `apiCallId` in multiple `assistant.message` events:** Retries, corrections, or streaming re-attempts may emit the same API call ID. Without dedup, the same call is counted multiple times.

2. **Per-turn vs. aggregate mismatch:** Sum of `outputTokens` across all `assistant.message` events may diverge from `session.shutdown.data.usage.outputTokens` due to:
   - Missing events (e.g., timeout before event write)
   - Cache token interaction (cache hits are real cost but may not appear in per-turn events)
   - Rounding or aggregation differences

3. **Multiple session loads:** If the reader is invoked multiple times (e.g., incremental updates), the same session must not be counted twice.

### Token Types
The data includes five distinct cost metrics:
- `inputTokens` — tokens sent to model
- `outputTokens` — tokens generated by model
- `cacheReadTokens` — tokens read from cache (real cost, charged by provider)
- `cacheWriteTokens` — tokens written to cache (real cost, charged by provider)
- `reasoningTokens` — tokens used in reasoning mode (subset of output, real cost)

Each must be tracked separately; combined totals should sum these without double-counting.

## Decision

### 1. Source Abstraction

**Unified Source, with IDE/CLI Extensibility**

Define a `Source` enum:
```
enum Source {
  Unknown,     // fallback for unparseable events
  Session      // current CLI + IDE (both write to same schema)
}
```

**Rationale:**
- Today, CLI and IDE use the same event schema and are structurally identical.
- No separate `host_type` or `environment` marker exists in event data.
- Treating them as unified simplifies the reader and avoids premature branching.
- If future IDE integration adds a distinct signal (e.g., vscode event marker), extend the enum and add conditional logic at the reader entry point (not in dedup or normalization).

**Futureproofing:**
- Store `source: Source` in the normalized record.
- If IDE later adds `vscode.metadata.json` or environment marker in session start, add detection logic that sets `source = Source::IDE` and log the split for audit.

### 3. Dedup Key (Concrete Implementation)

**Primary Dedup Key:**
```
{sessionId}:{eventId}
```

**Format Example:**
```
sess_abc_123:evt_def_456
```

**Algorithm:**
```
dedupSeen := make(map[string]bool)
for each event in session.events {
  key := "{event.parentId}:{event.id}"
  if dedupSeen[key] {
    log.warning("duplicate event detected", key)
    continue  // skip this event
  }
  dedupSeen[key] = true
  // process event
}
```

**Scope:** Per session (ensures uniqueness within a session).

**Why this key?**
- `sessionId` (event.parentId) + `eventId` (event.id) together are globally unique (SDK responsibility).
- Eliminates event deduplication from network retries or repeated file reads.
- O(1) set membership check; scales to millions of events.

---

### 4. Secondary Dedup: apiCallId Duplication

**Risk:** Same `apiCallId` may appear in multiple `assistant.message` events (retries, corrections, or streaming retransmission).

**Rule:** Group by `apiCallId` within a session. If multiple turns share the same `apiCallId`:
1. Keep the **earliest by timestamp** (most likely the primary attempt).
2. Log all others as warnings (audit trail for investigation).
3. Sum only the kept turn's tokens.

**Algorithm:**
```
apiCallGroups := make(map[string][]Turn)
for each turn in session.turns {
  if turn.api_call_id != nil {
    apiCallGroups[*turn.api_call_id] = append(apiCallGroups[*turn.api_call_id], turn)
  }
}

for apiCallId, group := range apiCallGroups {
  if len(group) > 1 {
    // sort by timestamp ascending
    sort.Slice(group, func(i, j int) bool {
      return group[i].timestamp < group[j].timestamp
    })
    log.warning("duplicate apiCallId detected", apiCallId,
               "keeping earliest at", group[0].timestamp,
               "discarding", len(group)-1, "later attempts")
    // keep group[0]; discard others
  }
}
```

### 5. Tertiary Dedup: Per-Turn vs. Aggregate Reconciliation

**Risk:** Sum of `outputTokens` from all `assistant.message` events may diverge from `session.shutdown.data.usage.outputTokens`.

**Rule:** Compute reconciliation status per session:
1. Sum all per-turn token counts across all `Turn` records.
2. Compare to aggregate token counts from `session.shutdown`.
3. If divergence is **≤±1%**, mark as `Ok`.
4. If divergence is **>±1%**, mark as `Divergence` and log warning.
5. If no `session.shutdown` event exists, mark as `MissingShutdown`.

**Formula:**
```
percentDiff(expected, actual) = abs(expected - actual) / expected * 100
if percentDiff <= 1.0:
  status = Ok
else:
  status = Divergence(percentDiff, "per-turn sum diverges from shutdown aggregate")
```

**Algorithm:**
```
perTurnSum := TokenCount{
  InputTokens:        0,
  OutputTokens:       0,
  CacheReadTokens:    0,
  CacheWriteTokens:   0,
  ReasoningTokens:    0,
}

for each turn in session.turns {
  perTurnSum.InputTokens += turn.input_tokens
  perTurnSum.OutputTokens += turn.output_tokens
  perTurnSum.CacheReadTokens += turn.cache_read_tokens
  perTurnSum.CacheWriteTokens += turn.cache_write_tokens
  perTurnSum.ReasoningTokens += turn.reasoning_tokens
}

if len(session.aggregates) == 0 {
  session.ReconciliationStatus = MissingShutdown
} else {
  for _, agg := range session.aggregates {
    outputDiff := percentDiff(perTurnSum.OutputTokens, agg.OutputTokens)
    inputDiff := percentDiff(perTurnSum.InputTokens, agg.InputTokens)
    
    maxDiff := max(outputDiff, inputDiff, ...)  // check all token types
    
    if maxDiff > 1.0 {
      session.ReconciliationStatus = Divergence{maxDiff, "..."}
      log.warning("reconciliation mismatch", session.Id, 
                 "per-turn sum diverges by", maxDiff, "%")
    } else {
      session.ReconciliationStatus = Ok
    }
  }
}
```

### 6. Precedence Rule: Ground Truth

**Hierarchy:**
1. **`session.shutdown` aggregate is ground truth** for final token counts (used for billing/reporting).
   - Rationale: Emitted once at session end; represents SDK's final authoritative tally.
   - Use: Final reports, billing reconciliation, SLA compliance.

2. **Per-turn `assistant.message` events are detail-level truth** (used for per-turn attribution and debugging).
   - Rationale: Subject to event buffering, network delays, duplicate retries.
   - Use: Per-turn cost attribution, turn-by-turn performance analysis, data quality investigation.

3. **When reconciliation diverges >1%:**
   - Report both values in output.
   - Flag session for manual review (logged separately).
   - Use aggregate for official tallies; use per-turn breakdown for investigation.

### 8. Local-Only Constraint — Verification

**Data Flow (All Local):**
```
Source 1: ~/.copilot/session-state/{sessionId}/events.jsonl
          ↓ (read via filesystem)
       [In-Memory Dedup]
          ↓
Source 2: ~/.copilot/session-store.db
          ↓ (read via SQLite driver)
       [Join on session_id]
          ↓
      [Normalize]
          ↓
      [Reconcile]
          ↓
      [Report] (file or stdout)
```

**No Network Calls:**
- ✅ No schema registry lookup
- ✅ No remote dedup service
- ✅ No API validation or authentication
- ✅ No telemetry upload during read
- ✅ No external database connection

**Dedup happens entirely in-memory:**
- Session-scoped dedup sets (sessionId:eventId)
- apiCallId grouping (per session, per model)
- Reconciliation comparison (per session)
- All computed before output

## Rationale

### Why Unified Source?
- **Fact:** CLI and IDE today use identical event schema (no host_type marker).
- **Risk:** Creating separate codepaths now would introduce branching that won't be tested until IDE actually ships.
- **Safe:** Store `source: Source` field; extend the enum when IDE data arrives. Current code path is "Source::Session".

### Why Primary Dedup on `{sessionId}:{eventId}`?
- **Simplicity:** Single, deterministic key; no sorting or complex logic.
- **Correctness:** Event IDs are unique by design (SDK responsibility); combining with session scopes them.
- **Efficiency:** O(n) set membership check; no multi-step correlation.

### Why Secondary Dedup on `apiCallId`?
- **Observed Risk:** Provider retry patterns can emit the same API call ID multiple times.
- **Safe Handling:** Keep earliest event (most likely primary); log others as audit trail.
- **Traceability:** Operators can investigate anomalies without silent data loss.

### Why Tertiary Dedup on Per-Turn vs. Aggregate?
- **Data Quality Signal:** Divergence flags transmission loss, timing skew, or cache timing.
- **1% Threshold:** Balances false positives (rounding) with true negatives (real data loss).
- **Dual Reporting:** Report both values; let downstream decide. Prefer aggregate if reconciliation Ok.

### Why Prefer `session.shutdown`?
- **Finality:** Emitted once at session end; represents SDK's final tally.
- **Per-Turn Risk:** Subject to event buffering, network delays, duplicate retries.
- **Trade-Off:** Lose per-turn detail in favour of aggregate accuracy. Per-turn events still available for breakdown reporting.

### Why Local-Only is Safe?
- **Data Volume:** Session files are small (<10MB each); in-memory dedup scales to thousands of sessions.
- **Correctness:** Dedup keys are computed deterministically; no external system can contradict them.
- **Privacy:** No telemetry about the read is sent anywhere; all computation is local.
- **Resilience:** No network timeouts, no service degradation; reader is always available.

---

## Consequences

### Positive
1. **Zero Dependency Failures:** No network calls means no timeouts, no API key mismanagement, no outage coupling.
2. **Privacy:** All processing stays on-device; no telemetry about usage patterns leaves the machine.
3. **Offline Capability:** Reader works without internet connectivity.
4. **Auditability:** Dedup keys and reconciliation flags provide clear audit trail for data quality investigations.

### Trade-Offs
1. **Per-Turn Detail Loss:** If reconciliation diverges >1%, we flag it but still report aggregate. Operators must investigate manually if per-turn detail is critical.
2. **Duplicate Detection (apiCallId) is Best-Effort:** If the same `apiCallId` appears with different timestamps in different sessions, dedup will not catch it (by design, scoped to session). This is correct but requires care if future analysis crosses session boundaries.
3. **Maintenance Burden:** If future token types are added by the provider (e.g., `toolCallTokens`), the canonical form must be updated. Not a blocker; just requires review.

### Operational Implications
1. **Monitoring:** Track per-session reconciliation status. Alert if >5% of sessions have divergence >1%.
2. **Manual Review:** Sessions flagged with divergence should be logged separately for periodic audit.
3. **Versioning:** If event schema changes, reader must handle multiple schema versions (e.g., old events missing `reasoningTokens`). Treat as optional fields.

## Alternatives Considered

### Alternative 1: Separate CLI and IDE Readers

**Approach:** Branch at the entry point; CLI reader uses Session source, IDE reader uses distinct source.

**Rejected Because:**
- **Premature:** IDE is not yet integrated; data is structurally identical today.
- **Duplicate Code:** Would require two parallel implementations of normalization, dedup, reconciliation.
- **Testing Gap:** IDE reader cannot be tested until IDE actually ships; dead code risk.
- **Future Flexibility:** If IDE adds a marker field, extending the unified reader is simpler than merging two codebases.

### Alternative 2: Trust Per-Turn Events Only (No Aggregate)

**Approach:** Ignore `session.shutdown` aggregate; compute all totals from per-turn `assistant.message` events.

**Rejected Because:**
- **Loss of Ground Truth:** Provider's final tally is in `session.shutdown`; per-turn events are lossy (may be buffered, dropped, or duplicated).
- **Cache Token Risk:** Cache tokens may only appear in aggregate, not per-turn events.
- **Retry Double-Counting:** Without secondary dedup on `apiCallId`, retries would inflate counts.
- **Incomplete:** `totalNanoAiu` (cost metric) only appears in aggregate.

### Alternative 3: Remote Dedup Service

**Approach:** Send all events to a central dedup service; service issues canonical dedup IDs.

**Rejected Because:**
- **Network Dependency:** Violates local-only constraint.
- **Privacy Risk:** Usage patterns leave the machine; no GDPR/compliance guarantee.
- **Latency:** Reader must wait for service; offline use impossible.
- **Scaling:** Service becomes bottleneck; adds operational burden.
- **Not Justified:** In-memory dedup handles today's volume; no evidence of need.

### Alternative 4: Trust Aggregate Only (No Per-Turn Validation)

**Approach:** Skip per-turn events; only read `session.shutdown`.

**Rejected Because:**
- **Loss of Detail:** Cannot answer "which turn used the most tokens?" or "model timeline."
- **Cache Interaction Blindness:** Cannot correlate cache hits with API call patterns.
- **Debugging:** Per-turn events crucial for troubleshooting model performance.

## Re-Evaluation Triggers

Review this ADR if:
1. **Event schema changes:** New token types added (e.g., `toolCallTokens`), or existing fields removed.
2. **IDE integration ships:** Verify whether IDE emits to same schema or separate source; if separate, update Source enum and entry-point branching.
3. **Provider changes retry behavior:** If `apiCallId` duplicates become systematic (>10% of events), reconsider secondary dedup or escalate to provider.
4. **Reconciliation divergence becomes endemic:** If >20% of sessions have >1% divergence, investigate whether per-turn events are being systematically dropped and consider fallback strategies.
5. **Performance constraints arise:** If >100k sessions need processing, verify in-memory dedup set fits in RAM; if not, implement tiered dedup (e.g., per-day buckets).
